Grounding Context

핵심 긴장: oh-my-skills는 skills 중심 표준(.claude/skills/{category}/{skill}/ + SKILL.md·README.md·GUIDELINES.md + frontmatter context/agent/language)이고, obsidian-workflows는 commands 중심 모델(commands/가 정본, hook 경로 commands/)로 의도적으로 갈라져 있습니다. 따라서 평가 결과는 (a) 순수 미준수 갭과 (b) 의도적 분기 두 종류로 나오며, 후자는 "맞추기"가 아니라 "분기를 명시 문서화"가 답일 수 있습니다.
Codebase Context — obsidian-workflows 실측 (검증됨)
  • skill name이 디렉터리와 전부 불일치: skills/plan/SKILL.md의 name: workflow-plan-reference (디렉터리는 plan). 4개 모두 workflow-*-reference 패턴.
  • SKILL.md는 수동 미러: 본문이 직접 "이 문서는 commands/ow/plan.md의 mirror이며, 갈리면 command가 canonical"이라고 선언.
  • frontmatter 스키마 분기: skill은 name/description/user-invocable/version/created/updated — oh-my-skills 필수 context/agent/language가 전부 없음.
  • 99바이트 "stub" 파일의 정체: skills/plan/plan.md 등은 tags: MOCs + folder-index-content — Obsidian Folder Index vault 노트. 저장소 자체 규칙 "Do not mix user vault content into this repository"를 위반.
  • skill 검증 사각지대: tools/check-frontmatter.sh는 find commands만 스캔 — SKILL.md는 10개 validator 중 어느 것도 건드리지 않음. 위 불일치가 CI를 멀쩡히 통과.
  • command 14개는 frontmatter 일관. generate-index.sh가 COMMANDS.md를 frontmatter에서 자동 생성(derive-not-handwrite 패턴 이미 보유). .claude-plugin/marketplace.json 존재, plugin.json은 최소(author/license 없음).
  • 이 저장소는 Claude Code 이벤트 훅을 쓰지 않음(상태머신 + Skill 도구 오케스트레이션).
평가 기준 — oh-my-skills 작성 규칙 (요지)
  • SKILL.md frontmatter 필수: name, description, version, context(fork|inline), agent, language. name은 kebab-case이며 디렉터리명과 일치. description ≤100자.
  • 레이아웃 skills/{category}/{skill}/ + SKILL.md + README.md + GUIDELINES.md (REFERENCE.md 선택).
  • scripts/가 있으면 scripts/tests/run.sh 필수 (BATS→Python→Shell, mktemp+HOME 격리+PATH stub).
  • hook command/path는 .claude/skills/로 시작. 신규 skill은 create-skill.sh로 스캐폴딩. semver + CHANGELOG.
  • 한국어 콘텐츠 규칙: 본문 한국어/식별자 영어, em-dash 금지(~·괄호), 외래어 표기법, 용어 일관성.
External Context — 공개 Agent Skills 표준 (배경)
  • Anthropic/agentskills.io 공개 표준은 name+description만 필수(name ≤64자, claude/anthropic 예약어 금지; description ≤1024자, 3인칭, "무엇을+언제"+트리거 키워드). 나머지는 관례.
  • SKILL.md ≤500줄 권장, references는 1단계 깊이, progressive disclosure. context: fork·hooks는 Claude 전용(비호환).
  • → 사실상 3계층: 공개 표준 ⊂ oh-my-skills 하우스 규칙. obsidian-workflows는 공개 표준은 충족하나 oh-my-skills 엄격층에서 의도적으로 분기.
Past Learnings
  • skill 작성/스펙 정합성 관련 학습은 docs/solutions/에 없음(미포착 영역). 기존 2건은 command 런타임 인시던트뿐 → 이 작업 결과는 /ce-compound로 포착 가치 있음.

Idea 1 · Top pick

정합성 stance를 먼저 선언하라 — 등급제 + 분기 레지스터

구조·패키징 분기 external + reasoned 확신 90% 복잡도 Low-Med
설명"이 저장소가 oh-my-skills에 맞춰야 한다"는 전제 자체를 먼저 결정한다. oh-my-skills 각 규칙을 adopt(맞춘다) / divert(사유 있는 분기) / n-a(해당 없음)로 태깅한 docs/conformance.md 레지스터를 만들고, 정합성을 이진 합/불이 아니라 등급(공개표준 A / oh-my-skills AA)으로 본다. 레지스터에 없는 분기는 "드리프트(버그)", 레지스터에 있는 분기는 "승인된 예외"로 취급한다.
근거external: WCAG A/AA/AAA 등급 정합성 + 규제 감사의 "예외 레지스터(통제+증거+사유)" 패턴. reasoned: 공개 표준은 name+description만 요구하므로 oh-my-skills 규칙 다수는 하우스 의견 — 이 저장소는 commands 중심을 의도 선택했다(CLAUDE.md가 commands/를 canonical로 명시).
의의stance가 없으면 나머지 6개 아이디어가 "갭"인지 "의도 분기"인지 매번 재논쟁된다. 레지스터는 그 재논쟁을 1회 결정으로 바꾼다 — 다른 모든 아이디어의 의미를 규정하므로 가장 먼저.
단점
레지스터를 만들어도 갱신하지 않으면 또 다른 stale 문서가 된다. divert 항목마다 검토일을 박고, Idea 2 validator가 "레지스터에 없는 분기"를 잡도록 묶어야 살아있는 문서가 된다.
Idea 2

skill-frontmatter validator를 CI 게이트로 — 지금 SKILL.md는 검증 사각지대

테스트·검증 패리티 direct 확신 92% 복잡도 Med
설명기존 check-frontmatter.sh의 형제로 check-skill-frontmatter.sh를 추가해 같은 pre-commit/CI 머신에 태운다. Idea 1에서 정한 스키마(어느 필드가 필수인지, name↔디렉터리 일치, description 길이)를 읽어 모든 현재·미래 skill을 머지 전에 강제한다.
근거direct: check-frontmatter.sh의 대상은 find commands뿐 — SKILL.md는 10개 validator 어디에도 안 걸린다(검증함). 그래서 name: workflow-plan-reference(≠디렉터리)가 오늘 CI를 통과한다.
의의검증기가 없으면 어떤 결정(맞추기든 분기든)도 강제되지 않아 시간이 지나며 조용히 표류한다. stance(Idea 1)를 정해도 게이트가 없으면 종이 위 규칙일 뿐 — 이게 enforcement 기반이다.
단점
게이트 도입 시 기존 4개 skill이 즉시 빨간불 → 게이트 도입과 위반 정정을 같은 커밋으로 묶어야 한다(이 세션에서 이미 겪은 gate-ordering 패턴).
Idea 3

skill↔command 미러를 해소 — SKILL.md를 command에서 생성

Skill 파일·레이아웃 direct + reasoned 확신 85% 복잡도 Med-High
설명각 트랙이 skills/{track}/SKILL.md와 commands/ow/{track}.md 두 곳에 손으로 존재하고, SKILL.md가 스스로 "command의 mirror"임을 선언한다. generate-index.sh가 이미 COMMANDS.md를 frontmatter에서 생성하듯, SKILL.md(+oh-my-skills frontmatter)를 command 정의에서 생성하게 만든다. 손으로 두 번 쓰지 않으니 표류 클래스 자체가 사라진다.
근거direct: skills/plan/SKILL.md 본문이 "이 문서는 commands/ow/plan.md의 mirror... command가 canonical"이라 명시 + CLAUDE.md "single canonical source per command name". reasoned: 수동 미러는 정의상 표류하고, 이 저장소는 최근에도 stale 참조 수정 커밋이 반복됐다.
의의생성으로 바꾸면 Idea 4의 frontmatter 필드(context/agent/language)도 생성기가 주입할 수 있어 4개 파일 손편집이 사라진다 — 정합성이 빌드 단계가 된다.
단점
생성기는 command→SKILL 매핑 규칙(어떤 본문을 어떻게 요약할지)을 설계해야 해 초기 비용이 가장 크다. 대안: 생성 대신 "drift-check"(미러가 갈렸는지 CI 검출)만 먼저 두는 더 가벼운 변형.
Idea 4

frontmatter 정정 — name↔디렉터리 일치 + context/agent/language는 정직하게 결정

Frontmatter 정합성 direct + external 확신 88% 복잡도 Low-Med
설명두 가지를 정정한다. (1) name을 디렉터리와 일치시킨다 — 현재 workflow-plan-reference≠plan. (2) 누락된 context/agent/language는 채우기 전에 결정: 이 skill들이 실제로 fork 에이전트로 도느냐 인라인 엔트리포인트냐를 보고 정직하게 context: inline을 선언하거나, "우리 스키마(user-invocable 등)가 정본"임을 Idea 1 레지스터에 divert로 기록한다. 런타임이 안 읽는 agent 필드를 cargo-cult로 박지 않는다.
근거direct: 4개 skill 모두 name: workflow-*-reference로 디렉터리 불일치(검증함) + context/agent/language 부재. external: 공개 표준은 이 셋을 요구하지 않음 — oh-my-skills 엄격층만 요구하므로 "채울지/분기할지"는 진짜 선택지다.
의의name↔디렉터리 매핑이 사람 암묵지에 의존하면 skill 추가·이동마다 조용한 불일치가 쌓인다. 일치는 자동 강제 가능한 규칙이라 비용 대비 효과가 크다. 필드 결정은 "정합성 연극"을 피하는 핵심이다.
단점
name 변경은 외부에서 이 skill을 부르는 참조(plugin/marketplace 인덱스)를 건드릴 수 있어, 변경 전 참조 영향 확인이 필요하다.
Idea 5

skill 디렉터리의 Obsidian MOC vault 파일 제거 — 저장소 자체 규칙 위반

Skill 파일·레이아웃 direct 확신 95% 복잡도 Low
설명각 skill 디렉터리의 99바이트 plan.md/work.md 등은 skill 산출물이 아니라 tags: MOCs + folder-index-content를 담은 Obsidian Folder Index vault 노트다. 제거하거나(또는 stance에 따라 정당한 README/GUIDELINES로 대체) 정리한다.
근거direct: skills/plan/plan.md 내용 확인 — tags: MOCs + folder-index-content 블록. 저장소 CLAUDE.md "Do not mix user vault content into this repository"를 정면 위반.
의의vault 전용 파일이 plugin core에 커밋돼 있으면 (a) 자기 규칙 위반이고 (b) skill 레이아웃을 오독하게 만든다("이게 README 자리인가?"). 가장 명확하고 값싼 정리 — 즉시 가능한 cleanup.
단점
이 파일들이 누군가의 Obsidian vault 뷰(Folder Index 플러그인)에서 실제로 쓰이는지 먼저 확인해야 한다. plugin 저장소엔 불필요하지만, 저장소가 vault와 겸용된다면 .gitignore 처리가 맞을 수 있다.
Idea 6

스캐폴딩 + skills 인덱스 생성 — 결함을 태어날 때 막는다

테스트·검증 패리티 direct 확신 80% 복잡도 Med
설명두 가지 도구를 이 저장소 컨벤션에 맞춰 도입한다. (1) create-skill.sh 스캐폴더 — 새 skill을 올바른 frontmatter·레이아웃으로 태어나게 해 Idea 4·5의 결함 재생산을 원천 차단한다. (2) generate-skills-index — 이미 검증된 generate-index.sh(COMMANDS.md 자동 생성) 머신을 skills/*/SKILL.md로 확장해 SKILLS.md를 생성한다.
근거direct: generate-index.sh가 이미 frontmatter→COMMANDS.md derive 패턴을 가동 중(검증함). oh-my-skills는 create-skill.sh + generate-skills-index를 요구하나 이 저장소엔 skill 스캐폴딩이 없다.
의의스캐폴딩은 "정합 경로 = 최소저항 경로"로 만드는 최고 레버리지 수: 이후 모든 skill이 태어날 때부터 정합이라 Idea 2 validator가 거의 빨간불을 안 낸다. 인덱스 생성은 검증된 파서를 재사용하므로 거의 공짜.
단점
Idea 3(생성형 SKILL.md)을 채택하면 스캐폴더와 역할이 겹친다 — 두 도구의 경계(생성 vs 수기 시작점)를 먼저 정해야 중복 투자를 피한다.
Idea 7

한국어 콘텐츠 + description lint — 용어집·em-dash·길이·트리거

한국어 콘텐츠·문서 품질 external + direct 확신 78% 복잡도 Med
설명기존 lint-frontmatter/validate-markdown를 확장해 공유 규칙 세트를 skill·command 14개에 동일 적용한다: description ≤상한·3인칭/트리거 키워드(Anthropic 스타일), em-dash 금지, 외래어 표기법, 단일 용어집 파일 기준 용어 일관성. 용어집 한 번 편집이 저장소 전체 한국어 표면을 재지배한다.
근거direct: 저장소에 lint-frontmatter·validate-markdown 존재 + plan/SKILL.md description이 100자 초과·트리거 키워드 부재. external: Anthropic은 description 3인칭+"무엇을/언제"+트리거를 요구; oh-my-skills는 ≤100자·외래어 표기법·용어 일관성을 요구.
의의description은 skill 자동 트리거 정확도를 좌우하는 유일한 메타데이터다. em-dash는 팀이 싫어하는 대표적 AI 문체 흔적이라 하드 차단 가치가 있다. 용어집은 저자가 바뀌어도 용어가 갈라지지 않게 한다.
단점
한국어 형태소·외래어 검사를 정규식/접미사 사전으로 근사하면 오탐이 난다. 길이·em-dash 같은 결정적 규칙부터 하드 게이트로 걸고, 외래어/용어는 경고 수준에서 시작하는 게 안전하다.
