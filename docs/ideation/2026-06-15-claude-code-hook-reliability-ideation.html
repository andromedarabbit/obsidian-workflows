Grounding Context

Codebase Context

obsidian-workflows is a state-machine driven Claude Code plugin for Obsidian writing workflows. Orchestration is via .claude/state/*.json — the repo does not currently use hook-event subscriptions.

  • Existing L1 static validators: check-frontmatter.sh, validate-command.sh, validate-hook-paths.sh
  • L2 behavior contract validator (docstring-based); 5 manual smoke-test specs (.md)
  • No stdin contract validators; no automated hook test runner
  • Stale-reference fix previously committed as 532c3e5 (hook contract / stale reference recovery)
User-Named Constraint: claude-code-hook-playbook-2026-06-15.md

Hard-won lessons from an oh-my-skills session. Four documented silent-failure classes:

  • matcher matches tool name strings only (Bash, Edit, Write…) — not content or file paths. Content filtering belongs in the if field or in-script stdin parsing.
  • PostToolWrite is a non-existent event. The 29 official events are enumerated in SchemaStore.
  • $FILE env var doesn't exist. Claude Code passes context via stdin JSON: tool_input.file_path, tool_input.command, hook_event_name, tool_name, session_id, cwd, transcript_path.
  • No official SKILL.md hook validator. Testing pyramid: L1 static lint (no claude) → L2 stdin injection (PATH stub + HOME isolation) → L3 real claude headless.
  • Pre-commit gate ordering: introduce validator + fix violations in the same commit; .pre-commit-config.yaml must be in the first commit.
Past Learnings
  • Canonical single source of truth prevents drift across documentation and validators.
  • fd (not git log) for filesystem queries in hook scripts — 10-100× faster.
  • Scenario-based fixture tests catch regressions that docstring checks miss.

Topic Axes

1Hook authoring correctness— preventing matcher/event/stdin mistakes at write time

5Developer feedback loop— how fast/clearly does a developer know a hook is broken?

Idea 1

Hook Validator Suite

CI/CD integration + Hook authoring correctness direct: Confidence 95% Complexity Medium
Description A comprehensive static linter that checks every hook definition for three silent-failure classes simultaneously: (a) non-existent event names against the 29-event SchemaStore enum, (b) matchers that regex-match no known tool name string, and (c) if field expressions on non-tool events. Runs in CI as a hard block with SARIF-compatible output for inline PR annotations. No claude binary required — stdlib Python/bash only.
Basis direct: Playbook §Problem 1 ("Bash.*git.*commit → 절대 발동 안 함 ❌"), §Problem 2 (PostToolWrite = dead hook, full 29-event list from SchemaStore), §Problem 4 (validate-matchers.sh pattern). SchemaStore JSON schema provides the authoritative event enum at no cost.
Rationale Silent failure is the dominant failure mode: a wrong event name or dead matcher produces zero errors, zero output, no signal — for months. A static gate that runs on every commit closes this entire class before it reaches production, with no runtime dependency.
Downsides
Introducing the gate while violations exist blocks CI immediately — gate-ordering discipline required (introduce + fix in same commit, per playbook §Git B). Tool-name list needs updating when Claude Code adds new tools.
Idea 2

Canonical Event Enum as Single Source of Truth

Hook authoring correctness direct: Confidence 90% Complexity Low
Description Extract the 29 official Claude Code hook events and the set of known tool name strings into a single machine-readable file (config/hook-events.yaml). Every downstream artifact — the Hook Validator Suite, IDE autocomplete snippets, documentation tables, and test fixture templates — reads from this file. When Anthropic adds or renames an event, one change propagates everywhere.
Basis direct: Playbook §Problem 2 identifies SchemaStore (schemastore.org/claude-code-plugin-manifest.json) as the authoritative event enum source and documents how drift caused 10+ silent broken hooks. reasoned: A single checked-in enum file eliminates the class of drift bugs where validators, docs, and snippets each maintain their own copy of the 29-event list.
Rationale The broken hooks in the playbook's intro were caused by knowledge drift — developers used event names they believed were correct without a single source to check against. Making the enum a checked-in artifact turns it from implicit knowledge into enforced reality, and every future tool can rely on it without re-fetching SchemaStore.
Downsides
Requires periodic synchronization with SchemaStore when Claude Code releases update the event set. Needs a CI check confirming the file is sourced from SchemaStore rather than diverging silently.
Idea 3

Hook Template Generator (tools/new-hook.sh)

Knowledge propagation direct: Confidence 85% Complexity Low
Description A scaffolding script that generates a correctly-structured hook script given an event name and tool name: shebang, stdin-guard preamble (detects no-arg invocation, parses tool_input.file_path via python3 -c), exit-contract inline comments (exit 0 = allow, exit 2 + stderr = block, other = non-blocking), and a clearly-marked # TODO: business logic section. Generated files pass L1 lint immediately, with no manual boilerplate to get wrong.
Basis direct: Playbook §Problem 3 documents the stdin guard pattern verbatim (the multi-case shell guard, the python3 -c 'import json,sys; ...' idiom, and the exit code semantics) but does not codify it as a generator. This is the playbook's content + the observation that no scaffold currently exists in tools/.
Rationale Every hook written from scratch passes through a danger zone: the author either knows the stdin schema and exit contract, or they don't. The generator makes the correct path the zero-effort path. Developers who copy-paste from the scaffold absorb correct patterns even without fully understanding why each line exists.
Downsides
Scaffold generators require maintenance as the stdin schema evolves. Developers may copy-paste and stop short of understanding the underlying contract — acceptable for productivity but limits knowledge transfer beyond the immediate task.
Idea 4

L2 Stdin Contract Test Harness

Validation and testing direct: Confidence 90% Complexity Medium
Description A minimal test runner (tools/test-hook.sh) that injects a fixture JSON payload via stdin into a hook script and asserts exit code, stderr, and optional stdout. Fixture files (tests/hooks/fixtures/) carry the exact stdin shape Claude Code sends: tool_name, tool_input, hook_event_name, cwd, session_id, transcript_path, permission_mode. PATH-stub + HOME-isolation ensures no test touches real external tools. npm run test:hooks runs all specs in under 5 seconds with no claude binary.
Basis direct: Playbook §Testing L2 documents the exact pattern: "Claude Code가 보내는 stdin JSON을 그대로 주입해 발화/no-op·출력계약 단언. 외부 CLI는 PATH stub, HOME은 mktemp로 격리." and provides the run_check() bash function verbatim. Codebase grounding confirms "no stdin contract validators, no automated E2E hook test runner" exist.
Rationale L1 (linting) catches structural mistakes at definition time. L3 (real Claude) catches runtime mistakes but is slow and has dependencies. L2 closes the middle: it detects logic bugs in stdin parsing, incorrect exit codes, and contract violations — in seconds, with zero runtime dependencies. Without it, a refactor that changes how a hook reads tool_input.file_path breaks silently in production.
Downsides
Fixture files must be maintained as the stdin schema evolves. Test coverage is only as good as the fixture library — hand-authored fixtures may miss edge cases Claude actually sends. Requires discipline to write fixtures alongside new hooks.
Idea 5

Hook Execution Audit Log

Developer feedback loop reasoned: Confidence 80% Complexity Medium
Description A lightweight shim wrapper (one sourced shell library line at the top of a hook script) that appends a structured JSON entry to ~/.claude/hook-audit.jsonl per invocation: timestamp, hook name, event, tool, exit code, duration. A companion tools/hook-audit-summary.sh surfaces hooks that have never fired across recent sessions. Activate verbose mode with HOOK_DEBUG=1.
Basis direct: Stdin JSON contains session_id, transcript_path, hook_event_name, and tool_name (playbook §Problem 3) — exactly what's needed for an audit record. reasoned: The playbook names the root cause of the bug class as lack of observation: "아무도 훅 발동을 관측하지 않았다." A shim that writes one JSON line per call costs near-zero overhead and converts invisible behavior into a searchable stream.
Rationale All other ideas prevent new bugs. The audit log diagnoses existing ones. Right now, "did my hook fire?" requires adding echo statements and re-running Claude. The log answers that question in one grep, retroactively, for any past session — without re-running anything. It turns months-silent failures into "hook X has never fired in 47 sessions."
Downsides
hook-audit.jsonl grows indefinitely without rotation (needs a configurable log path + max-size policy). The shim is opt-in unless the Template Generator (#3) includes it by default. In headless/CI environments, ~/.claude/ may not exist — needs a fallback log path.
Idea 6

Executable Runbook

Knowledge propagation direct: Confidence 85% Complexity Low
Description Convert the playbook into a permanent, CI-tested documentation artifact at docs/hook-authoring-guide.md. Every runnable code block — the matcher examples, the if field patterns, the stdin guard boilerplate, the L2 test harness usage — is annotated with a <!-- verify --> marker and validated by tools/test-docs.sh in CI. When a code block goes stale (e.g., the stdin schema changes), CI fails with a reference to the specific block.
Basis direct: The playbook identifies the root cause of the bug class as knowledge absence: "이 버그가 살아남은 단 하나의 이유 = 아무도 훅 발동을 관측하지 않았다." The playbook IS the knowledge — the gap is persistence and discoverability. Past learnings confirm "canonical single source of truth prevents drift" and scenario-based fixture tests are the right regression prevention pattern.
Rationale The playbook represents the hardest lessons from this session. Without a permanent, enforced home, this knowledge will be re-discovered in the next session that adds hooks. Making the guide executable (CI-tested code blocks) means future refactors that break the examples cause build failures, keeping the guide accurate indefinitely — not just accurate at the time of writing.
Downsides
tools/test-docs.sh needs careful scoping — not every code block is independently runnable (some require a vault or live Claude context). Start with the highest-value blocks (stdin guard template, event enum list, exit contract examples) and grow coverage incrementally.
Idea 7

Hook Registry Inversion

CI/CD integration direct: Confidence 75% Complexity Medium
Description Instead of manually maintaining hook entries in settings.json, a sync script (npm run sync:hooks) crawls all .md files in commands/ and .claude/skills/, extracts every hooks: frontmatter block, and writes the canonical hook list back to settings. The settings file becomes a derived artifact, not a hand-edited one. Stale references (pointing to deleted scripts) and missing registrations become structurally detectable — the sync script flags them on each run.
Basis direct: Commit 532c3e5 ("fix(contract): ow 명령 계약과 stale 참조 정합성 회복") demonstrates that stale references in the hook registry are a real, recurring issue in this repo. CLAUDE.md's stated principle "Keep command discovery deterministic (single canonical source per command name)" applies equally to hooks. reasoned: If settings is derived from skill files, the category of "settings points to a script that moved/was deleted" becomes detectable at sync time.
Rationale The 532c3e5 commit had to fix stale references manually. Registry inversion makes that class of fix structurally unnecessary: the sync script produces a diff when any script moves or disappears, surfacing the inconsistency at CI time rather than months later during a debug session.
Downsides
The sync script must parse all .md frontmatter reliably — YAML parsing in bash is fragile, prefer a Node.js script. Settings may have entries from other mechanisms (system-level hooks, team-shared config) that the sync script doesn't know about. Requires a policy decision: which entries are auto-synced vs. manually maintained.

Rejection Summary

# Idea Reason Rejected
— Event Name Validator (standalone) Duplicates stronger idea — merged into Hook Validator Suite (#1) which adds matcher and if: checks in the same tool
— Matcher Dead-Zone Detector (standalone) Duplicates stronger idea — merged into Hook Validator Suite (#1)
— if: Field Consistency Validator (standalone) Duplicates stronger idea — absorbed by Hook Validator Suite (#1)'s third check
— Fixture Auto-Generator Too expensive relative to value at this repo's scale; L2 Harness (#4) already solves the fixture problem with a small curated library
— Hook Liveness Canary Duplicates stronger idea — Audit Log (#5) provides never-fired detection; canary adds marginal value once the log exists
— Hook-to-Event Coverage Map (standalone) Below standalone ambition floor; naturally subsumed by the validator suite's SARIF output and any future reporting layer
— Self-bootstrapping Pre-commit Real but narrow — gate ordering is a one-time problem at introduction, not an ongoing concern warranting a dedicated tool
— Schema Drift Detector Too speculative — solves a future schema-version problem the current repo doesn't yet have; revisit when SchemaStore updates break something
— Migration-Gated Hook Upgrades (Flyway analogy) Too expensive — over-engineered for a single-team plugin repo; analogy is sound but the overhead doesn't match the scale
— Failure-Record Documentation (errors explain themselves) Absorbed by Hook Validator Suite (#1) — the validator should emit helpful "did you mean?" messages; not a standalone idea
— Inline stdin Schema Surfacing Duplicates — overlaps IDE integration and is absorbed by the Template Generator (#3) as standard scaffold commentary
— IDE Schema Integration (standalone) Derivative of Canonical Enum SSOT (#2); natural follow-on once the enum file exists — team can pursue at low marginal cost
— File-Watcher L1 Linter (save-time feedback) Below ambition floor — pre-commit gate is already adequate; save-time feedback is a nice DX improvement but not a distinctive standalone idea
— Self-asserting Hook Preamble Absorbed by Template Generator (#3) — the scaffold template includes this as standard boilerplate, which is simpler than opt-in per-hook preambles
— Hook Signal Shape Annotations (signal_shape:) Too expensive — introduces new YAML schema complexity with no existing tooling ecosystem to parse or enforce it
— Consumer-Driven Hook Contracts (Pact-style) Too expensive — appropriate for multi-team microservice contracts, overkill for a single-team plugin repo
— Script-as-Source-of-Truth Architecturally complex — inverting SKILL.md's role requires new tooling; Registry Inversion (#7) achieves a similar SSOT benefit with lower disruption
